# Code Review — 2026-04-03

Covers commits `df464fb..932cdb5`. The prior review flagged runner isolation, injection, updateTask atomicity, and the Bun.sleep test — all four are now fixed. Remaining and new issues are catalogued below.

---

## Strengths

- `runnerFactory: () => IAgentRunner` cleanly solves the shared-runner problem. Each `handleTaskAssigned` call creates its own isolated `ClaudeRunner`, matching the spec's "one runner instance per session" requirement.
- `handleUserComment` now correctly injects via `session.runner` (the per-task instance).
- `updateTask` builds a single parameterised UPDATE statement, making title+status patches atomic.
- `flush()` using `pendingEvents: Set<Promise<void>>` is a clean, deterministic replacement for `Bun.sleep(10)` with no observable side-effects in production.
- Test coverage is thorough and tests actual behavior against a real in-memory SQLite database — not just mocks.

## Issues

### Critical (Must Fix)

**`apps/orchestrator/src/orchestrator-api.ts:164`** — Error thrown inside the runner loop marks the task as `completed` instead of `failed`

When `runner.run()` throws a non-abort error, the `catch` block re-throws after emitting an error result. The `finally` block then runs `updateTask(task.id, { status: "completed" })` unconditionally. A task that crashed mid-run gets status `"completed"`, which is indistinguishable from success. Introduce a `"failed"` status or at minimum track and log the error path in `finally`.

**`apps/orchestrator/src/orchestrator-api.ts:93`** — `stop()` does not wait for runner loops to drain before it resolves

`stop()` calls `runner.stop()` on all active sessions, which aborts the `AbortController`, but resolves immediately while the `handleTaskAssigned` `for await` loop is still draining. On container shutdown (SIGTERM) the process can exit before the `finally { activeSessions.delete; updateTask(completed) }` blocks finish, dropping the DB write.

Fix: `stop()` should `await flush()` after stopping all runners.

---

### Important (Should Fix)

**`apps/orchestrator/src/main.ts:21`** — No graceful shutdown signal handler _(unfixed from prior review)_

No `SIGTERM`/`SIGINT` handler. On container shutdown the process is killed without running `orchestrator.stop()`, leaving active tasks stuck as `"active"` in the DB and SQLite WAL frames unsynced.

Fix:
```typescript
process.on("SIGTERM", async () => { await orchestrator.stop(); process.exit(0); });
process.on("SIGINT",  async () => { await orchestrator.stop(); process.exit(0); });
```

**`packages/claude-adapter/src/claude-runner.ts:116`** — `stoppedResult(stopped: boolean)` misleadingly named _(unfixed from prior review)_

The parameter is always `true` at the call site (abort path) or `false` (unreachable SDK contract violation path). Merging both into one method named `stoppedResult` obscures intent.

Fix: rename to `syntheticResult(finishReason: FinishReason)` and call it with the explicit reason. Remove the boolean parameter.

**`packages/claude-adapter/src/claude-runner.ts:126`** — `errorResult(_config: RunConfig)` takes an unused parameter _(unfixed from prior review)_

`_config` is never read. Remove the parameter: `private errorResult(): SessionResultMessage`.

**`apps/orchestrator/src/sqlite-task-store.ts:111`** — `updateTask` with empty patch silently succeeds _(unfixed from prior review)_

When `patch` is `{}`, the UPDATE is skipped but `getTask` is still called and the task returned. There is no test for this edge case and the semantics (no-op read vs. meaningful update) are undocumented.

Fix: add a test, and optionally add an early guard that throws or returns early with a documented no-op contract.

**`packages/claude-adapter/src/message-map.ts:152`** — `toolName: ""` on `ToolResultMessage` _(unfixed from prior review)_

The SDK does not carry the tool name on `tool_result` blocks so the adapter always emits `toolName: ""`. Any downstream consumer branching on `toolName` receives an empty string silently.

Fix: make `toolName` optional on `ToolResultMessage` in the core spec (`toolName?: string`), or document the limitation in the type definition.

**`packages/claude-adapter/src/index.ts:2`** — `toAgentMessages` exported from adapter public surface _(unfixed from prior review)_

`toAgentMessages` is an internal SDK-to-core mapping function. Exporting it leaks internal structure and invites callers outside the adapter to depend on it.

Fix: remove `toAgentMessages` from the package index. It is only used inside `claude-runner.ts`.

**`packages/claude-adapter/package.json:9`** — `@anthropic-ai/claude-agent-sdk` pinned with caret in adapter, exact in root _(unfixed from prior review)_

Root pins exactly (`"0.2.91"`), adapter uses caret range (`"^0.2.91"`). Violates the "dependencies must be version pinned" rule. A lockfile regeneration could resolve the adapter's copy to a different version, causing type drift.

Fix: change `"^0.2.91"` to `"0.2.91"` in `packages/claude-adapter/package.json`.

### Minor (Nice to Have)

**`apps/orchestrator/src/orchestrator-api.ts:210`** — `"cli"` string literal in three places instead of a typed constant _(unfixed from prior review)_

`taskRef.platform === "cli"` appears at lines 210, 223, 225, and 237. A future rename requires grep-and-replace across all sites.

Fix: extract a `const CLI_PLATFORM = "cli"` constant, or narrow `TaskRef` to a discriminated union as the spec implies.

**`apps/orchestrator/src/message-to-activities.ts:61`** — `formatToolDescription` hardcodes Claude Code tool names in the orchestrator layer _(unfixed from prior review)_

`Read`, `Bash`, `Write`, `Edit`, `Glob`, `Grep` are Claude Code-specific names in a layer that should be provider-agnostic. A different runner's tool names would fall through to the generic path silently.

Fix: move `formatToolDescription` to a registry that adapters contribute to, or document explicitly that Phase 1 is Claude-only.

**`packages/core/src/agent-runner.ts:3`** — `McpServerConfig` uses optional fields instead of a discriminated union _(unfixed from prior review)_

Both `command?` and `url?` are optional, allowing construction of an invalid config with both set, or neither set (silently skipped by `mapMcpServers`).

Fix:
```typescript
type McpServerConfig =
  | { type: "stdio"; command: string; args?: string[]; env?: Record<string, string> }
  | { type: "http"; url: string; headers?: Record<string, string> };
```

**`apps/orchestrator/src/orchestrator-api.ts:40`** — Two public entry paths with different async contracts _(unfixed from prior review)_

`runTask(event)` is awaitable; `handleEvent(event)` is fire-and-forget. Both call `handleTaskAssigned`. The spec only defines `handleEvent` — `runTask` is an undocumented extension that confusingly overlaps.

Fix: consider removing `runTask` and using `handleEvent` + `flush()` everywhere, or document `runTask` clearly in JSDoc.

**`apps/orchestrator/src/orchestrator-api.ts:56`** — `flush()` JSDoc says "For test use" but `stop()` needs it for correctness

Once `stop()` is fixed to call `flush()`, the "For test use" label is wrong. Remove the qualifier.

## Assessment

**Ready to merge:** No — with fixes required.

**Reasoning:** The four targeted fixes are all correct. However, the runner factory change exposed a new correctness gap in `stop()` (it signals shutdown but doesn't await drain), and five issues from the prior review remain unfixed — most notably the missing signal handlers in `main.ts` and the `toAgentMessages` adapter boundary leak.
